Commands Explained

int range <port-range> Selects a range of continuous interfaces and applies configuration to all of them at once. For example, int range f0/1-2 targets FastEthernet 0/1 and 0/2 simultaneously.

channel-protocol lacp Sets the EtherChannel negotiation protocol. Cisco supports two options:

  • LACP (Link Aggregation Control Protocol) — the IEEE 802.3ad standard protocol, works across different vendor devices.
  • PAgP (Port Aggregation Protocol) — Cisco proprietary, only works between Cisco devices.

LACP is generally the better choice unless you are in a purely Cisco environment and have a specific reason to use PAgP.

channel-group <number> mode active Adds the interface to an EtherChannel bundle and enables LACP in active mode, meaning the switch will actively send LACP packets to negotiate the bundle with the other end.

int port-channel <number> Enters the logical port-channel interface for further configuration such as setting the trunk mode and enabling PortFast.

The remaining commands such as switchport mode trunk and spanning-tree portfast are covered in detail in the Enterprise Networking section.

Why Use EtherChannel?

There are a few solid reasons why a network engineer would choose EtherChannel over just using a single link between switches.

Increased Bandwidth Instead of being limited to the speed of one physical link, EtherChannel combines the bandwidth of all bundled ports. Two FastEthernet links bundled together give you an effective 200Mbps instead of 100Mbps. This matters a lot on trunk links between switches carrying traffic for multiple VLANs.

Redundancy Without STP Blocking Normally if you connect two switches with more than one link, Spanning Tree Protocol will block all but one of those links to prevent loops. With EtherChannel, STP sees the entire bundle as a single logical link, so none of the physical ports get blocked. You get redundancy and full bandwidth at the same time.

Automatic Failover If one of the physical links in the bundle goes down, traffic automatically shifts to the remaining links with no disruption to the network. Users will not notice anything happened.

Load Balancing Traffic is distributed across all the physical links in the bundle based on a load balancing method such as source/destination MAC address or IP address. This makes better use of all available links rather than saturating a single one.

Potential Problems with EtherChannel

EtherChannel is useful but it can also cause headaches if not configured carefully.

Misconfiguration on One End EtherChannel requires both sides of the link to have matching configuration. If one switch has the channel-group set to active and the other is not configured for EtherChannel at all, the bundle will not form and you may end up with the ports being blocked by STP or flapping constantly.

Mismatched Parameters Both sides must match on speed, duplex, VLAN configuration, and trunk settings. If one port in the bundle is configured differently from the others, the entire EtherChannel can go down or behave unpredictably.

Load Imbalance Depending on the load balancing method configured, you might find that most traffic ends up flowing through just one or two links in the bundle while the others sit mostly idle. This defeats the purpose of bundling in the first place and requires tuning the load balancing algorithm to match your traffic patterns.

Troubleshooting Complexity When something goes wrong on a bundled link it can be harder to isolate which physical port is causing the issue. You have to look at both the port-channel interface and the individual member interfaces to get the full picture.
