Soc HomeLab

Security Operations Center Homelab 101

The goal of this lab is to build a functional SOC environment from scratch. Here is what we are working with:

  • SIEM: Wazuh
  • Firewall: OPNsense
  • Ubuntu Server (the target machine)
  • Attacker Machine: Kali Linux
  • Host Client: Ubuntu Client

The installation and setup process for each of these machines is straightforward and well documented, so I will skip that part and jump straight into what makes this lab interesting.

Why not Metasploitable? Most SOC homelabs use Metasploitable as the vulnerable target. I chose not to, because a machine that broken does not reflect anything close to a real environment. We are going to build our own attack surface instead.

Creating a Customised Attack Surface

The plan is simple:

  1. Stand up a database accessible over the network
  2. Exfiltrate data from the attacker machine
  3. Investigate the attack through Wazuh and harden the system

Setting Up the Environment

Run the following on your Ubuntu server to install the required packages:

sudo apt update
sudo apt install mysql-server python3 python3-pip
pip3 install flask mysql-connector-python

You will likely hit an error at the pip3 step as shown below.

This is caused by PEP 668, a Python enhancement proposal adopted by Debian and Ubuntu to prevent pip from modifying system-managed packages. The fix is to use a virtual environment instead.

python3 -m venv mysql_db
source mysql_db/bin/activate

Once activated, your terminal prompt will change to show you are inside the virtual environment.

Now run the pip3 command again and it will work fine.

Setting Up the Database

Start MySQL and create your database:

sudo mysql

CREATE DATABASE soc101;
USE soc101;

CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(100),
    password VARCHAR(100)
);

INSERT INTO users (username, password) VALUES ('admin', 'password123');

Feel free to add more users if you want a richer dataset to work with later.

Building the Backend

We need to expose this database over the network through a simple Flask application. This is what our attacker will be targeting.

from flask import Flask, request
import mysql.connector

app = Flask(__name__)

db = mysql.connector.connect(
    host="localhost",
    user="labuser",
    password="labpass",
    database="soc101"
)

@app.route('/login', methods=['GET'])
def login():
    username = request.args.get('username')
    password = request.args.get('password')

    cursor = db.cursor()
    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    cursor.execute(query)
    result = cursor.fetchall()

    if result:
        return "Login successful"
    else:
        return "Login failed"

app.run(host='0.0.0.0', port=5000)

Note for the sharp eyed: Yes, that query is intentionally vulnerable to SQL injection. That is the whole point. We want something to attack.

Access the service from your browser using:

http://server-ip:5000/login?username=admin&password=password123

If you run into errors along the way, read them carefully and troubleshoot individually. That is part of the process.

Configuring Wazuh to Collect the Right Logs

Just because your Ubuntu server is a Wazuh agent does not mean Wazuh is automatically collecting everything. Wazuh only monitors what you explicitly tell it to monitor. We need to configure the agent to capture logs from the database, the application, the filesystem, and system activity.

Replace the entire contents of /var/ossec/etc/ossec.conf on the agent with the configuration below. Make sure to update the server IP address to match your Wazuh manager.

Once you have the config in place, run the following commands on the agent to apply everything:

# 1. Backup current config
sudo cp /var/ossec/etc/ossec.conf /var/ossec/etc/ossec.conf.backup

# 2. Apply the new configuration
sudo nano /var/ossec/etc/ossec.conf

# 3. Enable MySQL general logging
sudo mysql -e "SET GLOBAL general_log = 'ON';"
sudo mysql -e "SET GLOBAL log_output = 'FILE';"
sudo mysql -e "SET GLOBAL general_log_file = '/var/log/mysql/mysql.log';"

# 4. Create MySQL log directory if it does not exist
sudo mkdir -p /var/log/mysql
sudo chown mysql:mysql /var/log/mysql
sudo chmod 750 /var/log/mysql

# 5. Restart the Wazuh agent
sudo systemctl restart wazuh-agent

# 6. Verify there are no errors
sudo tail -f /var/ossec/logs/ossec.log

Building the Wazuh Dashboard

Now that logs are flowing into Wazuh, we need to make sense of them visually. The goal is a real-time dashboard that shows:

  • Sudo privilege access attempts
  • Successful and failed SSH logins with usernames
  • Any access to /etc/passwd and /etc/shadow

Navigate to the Explore pane and open the Dashboards section.

I am running Wazuh version 4.14 for this walkthrough.

Click Create New and select the Pie Chart visualisation type.

Set the data source to Wazuh Alerts. Most of the configuration happens in the left panel:

Metrics tab Set the aggregation to Count and give it a custom name. I used “Privileged Alerts” but you can name it anything meaningful.

Buckets tab Set the aggregation to Terms, then choose your field. To know which fields are available and relevant, head over to the Discover section first and search for sudo. Open one of the events and look through the available fields to find what you need.

The field I used is rule.mitre.technique, which gives a nice breakdown of the techniques being triggered across your monitored endpoints.

From here we will build on top of this, adding metrics and tables to get a fuller picture of what is happening on the server. We will escalate the complexity gradually.
